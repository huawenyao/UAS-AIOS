# AI 招聘产品价值感设计

> 功能背后是业务目标，价值感来自「痛点是否被真正解决」以及「用户能否明显感知到」。本文档梳理业务目标、场景痛点、当前解决程度与价值感差距，并给出可落地的价值感设计建议。

---

## 一、业务目标与功能映射

| 业务目标 | 对应功能 | 当前实现 |
|----------|----------|----------|
| **提高初筛效率**（减少 HR 重复劳动） | 批量简历解析 + 多维度打分 + 排序 | ✅ 已实现（workflow_execution 批量处理） |
| **提升决策质量**（有据可依、可解释） | 推荐结论 + 证据链 + 风险标识 | ✅ 已实现（报告与 JSON 含 evidence/risk_flags） |
| **标准可配置、可审计**（合规与复盘） | 知识驱动权重 + 结构化存储 + 审计 | ✅ 已实现（evaluation_criteria、database、audit） |
| **面试准备与评估闭环**（初筛→面试→评价） | 候选人画像 + 面试维度 + 评价与通知 | ⚠️ 部分（画像与报告有；/evaluate、AI 面试完整闭环未落地） |
| **候选人体验与公平感**（透明、可反馈） | 可解释维度 + 证据链 + 公平性说明 | ⚠️ 部分（规则透明；无面向候选人的反馈通道与公平性呈现） |

功能对业务目标的**覆盖度不差**，但「是否真的解决痛点」和「用户是否有明显价值感」取决于**场景还原度**与**价值呈现方式**。

---

## 二、AI 招聘场景的真实痛点

### 2.1 HR 招聘专员

| 痛点 | 具体表现 | 当前是否真的解决 |
|------|----------|------------------|
| **海量简历看不过来** | 一个岗位几百份简历，人工逐份看耗时且易漏 | ✅ **部分解决**：批量扫描 + 打分排序，可先看推荐名单；但入口是脚本/命令行，HR 不会用则价值为 0 |
| **标准不统一** | 不同 HR、不同时间尺度不一，难以复盘 | ✅ **解决**：evaluation_criteria 统一维度与权重，可配置、可追溯 |
| **说不清为什么推荐/不推荐** | 业务或用人部门追问时只能凭印象 | ✅ **解决**：证据链与风险标识在报告中，可直接引用 |
| **发布岗位、收简历、跑筛选要多处操作** | 希望「一个地方搞定」 | ❌ **未解决**：无统一入口（Web/一键流程），需 Cursor 或改 JSON/跑脚本，**价值难以被感知** |

### 2.2 招聘经理/用人部门负责人

| 痛点 | 具体表现 | 当前是否真的解决 |
|------|----------|------------------|
| **看不懂推荐理由** | 只给一个分数或结论，不敢拍板 | ✅ **解决**：报告含推荐结论、风险标识、证据链（P0 已优化） |
| **担心漏掉合适的人** | 担心筛得太严或太松 | ⚠️ **部分**：有 borderline 档位与证据，但无「召回率/漏筛率」等可感知指标 |
| **需要快速拍板面试名单** | 时间紧，要一眼能决策 | ✅ **部分**：排序 + 档位清晰；若报告入口不显眼或需打开多份文件，**价值感打折扣** |

### 2.3 候选人（间接用户）

| 痛点 | 具体表现 | 当前是否真的解决 |
|------|----------|------------------|
| **投递后石沉大海** | 不知道有没有被看、为什么没下文 | ❌ **未解决**：系统无面向候选人的反馈（邮件/页面），规则再透明候选人也看不到 |
| **觉得筛选不透明、不公平** | 怀疑黑箱、偏见 | ⚠️ **部分**：规则与维度可解释、证据可追溯，但**候选人侧无任何触达**，公平感无法建立 |
| **希望得到改进建议** | 未通过时想知道差在哪 | ❌ **未解决**：无「未通过原因摘要」或「可对外反馈」的出口 |

### 2.4 面试官

| 痛点 | 具体表现 | 当前是否真的解决 |
|------|----------|------------------|
| **对候选人背景掌握不全** | 临时看简历，抓不到重点 | ✅ **解决**：报告中有画像、证据、风险，可支撑提问与评估 |
| **初筛与面试结论脱节** | 不知道初筛为什么推荐，面试不知如何对照 | ✅ **部分**：初筛证据可参考；面试后的结构化评估与初筛打通（/evaluate）**未完整落地**，闭环感不足 |
| **准备面试耗时长** | 希望系统给面试要点或提纲 | ⚠️ **部分**：证据链可当提纲；无「一键生成面试问题建议」等**显性省时**功能 |

### 2.5 HR 运营/配置管理员

| 痛点 | 具体表现 | 当前是否真的解决 |
|------|----------|------------------|
| **标准难调、难审计** | 改标准要改代码，审计无据 | ✅ **解决**：知识驱动配置 + 结构化存储 + 审计日志 |
| **多岗位、多部门标准不一** | 需要差异化权重或维度 | ✅ **支持**：通过配置与岗位维度可实现；若未在 UI/流程上暴露，**价值感偏后台** |

## 三、痛点解决度与价值感差距

### 3.1 真的解决了吗？

- **初筛效率、决策可解释、可配置可审计**：从能力上**已经解决**对应痛点，但有两个前提：
  - HR 能**用得上**（当前入口是脚本/Cursor，非技术 HR 用不上 → 效率价值为 0）。
  - 用户**看得见**（无「节省了多少时间」「推荐名单可解释率」等结果层面的呈现 → 价值感弱）。

- **候选人体验与公平感**：规则透明、证据可追溯，但**没有候选人侧的触达与反馈**，对候选人而言「没被解决」，公平感也无法建立。

- **面试闭环与面试官省时**：初筛到报告这一段**已解决**；从「面试完成 → 评价 → 与初筛对照」的闭环和「面试提纲/问题建议」等**显性省时**仍缺，价值感不完整。

### 3.2 为什么用户可能感觉不到明显价值？

| 原因 | 说明 |
|------|------|
| **价值未量化** | 没有「过去 100 份简历要看 2 小时，现在 5 分钟出推荐名单」的对比；没有「可解释推荐占比 100%」等结果指标。 |
| **入口与角色错位** | 最强价值受众是 HR，但入口是命令行/Cursor，HR 不会用；价值存在但**触达不了**。 |
| **价值未按角色呈现** | 同一份报告，HR、招聘经理、面试官关心的重点不同，若没有「为你角色总结的结论」或引导，容易觉得「只是又多了一份文件」。 |
| **候选人无触达** | 候选人看不到任何反馈与透明说明，企业侧再可解释，候选人也无法形成「这家公司筛选公平」的感知。 |
| **闭环未完成** | 初筛→面试→评价→通知若未在一套流程里走通，用户会认为「只是多了一个筛简历的工具」，而不是「招聘全流程助手」。 |

## 四、价值感设计建议（可落地）

### 4.1 让「效率」可被看见（HR / 招聘经理）

- **结果页首屏展示价值摘要**（可在现有 HTML 报告或后续 Web 中实现）：
  - 本批处理简历数、推荐/待定/不推荐数量；
  - 一句话：**「已为本次 X 份简历生成可解释推荐名单，推荐/待定共 Y 人，可直接用于面试名单决策。」**
- **可选：简单耗时统计**：记录「本批从上传/选择目录到报告生成」的耗时，与人工估算对比（如「约节省 2 小时」），在报告或摘要中展示。
- **入口收敛**：至少提供**一键式流程**（如单一 CLI：选目录 + 选岗位 → 跑匹配 → 打开报告），或简易 Web 上传 JD + 简历目录，让 HR 无需会命令行也能用上，**效率价值才能真实发生**。

### 4.2 让「决策依据」成为默认体验（招聘经理 / 面试官）

- **报告即决策页**：保持当前「推荐结论 + 风险标识 + 证据链」在首屏，不折叠；招聘经理打开报告就能答「为什么推荐/不推荐」。
- **面试官视图**（可在同一报告或单独页）：突出「建议关注的问题点」或「与岗位要求对照的缺口」，让「辅助面试准备」的价值一句话能说清。

### 4.3 让「候选人」有感知（公平与反馈）

- **对外可用的反馈话术**：从 evidence/risk_flags 生成「未通过原因摘要」（脱敏、合规），供 HR 在拒信或系统中使用，至少让**招聘方有能力**给候选人一个说法。
- **可选：候选人端页面/邮件**：如「您的简历已由 AI 辅助初筛，主要评估维度为……未通过原因概要：……」，把「可解释、有反馈」做成候选人可感知的触点。
- **公平性说明**：在对外说明或帮助页中写明「评估维度与权重公开、结果可审计」，并可选提供「公平性承诺」或抽样审计结论，增强信任。

### 4.4 让「闭环」与「结果」绑定（全角色）

- **流程可见**：在报告或工作台中标出当前处在「初筛完成 → 待面试名单 → 面试完成 → 评价已出」的哪一步，并让「下一步建议操作」明确（如：导出面试名单、发起 AI 面试、填写面试评价）。
- **事件与通知**：已实现的 Task/Event/Notification 在业务上用于「AI 面试完成 → 通知 HR/经理 → 生成评价」，并在界面上让用户看到「已通知」「已生成评价」，形成**完成感**。
- **一句话价值主张**（产品/运营可复用）：
  - HR：「批量简历 5 分钟出可解释推荐名单，再也不用逐份猜。」
  - 招聘经理：「每个推荐/不推荐都有证据和风险提示，决策有据可依。」
  - 候选人：「筛选标准透明，未通过也会得到可解释的反馈（在招聘方使用本系统并开启反馈时）。」
  - 面试官：「打开报告就有候选人画像和面试关注点，省掉自己翻简历的时间。」

### 4.5 优先级建议

| 优先级 | 动作 | 预期价值感 |
|--------|------|------------|
| P0 | 结果页/报告首屏增加「价值摘要」一句话 + 本批数量与推荐/待定人数 | 用户一眼知道「得到了什么」 |
| P0 | 提供 HR 可用的单一入口（Web 上传或一键 CLI），并配 30 字内说明 | 效率价值能真实触达 HR |
| P1 | 从 evidence 生成「未通过原因摘要」及对外使用指引（HR 复制到拒信/系统） | 候选人侧开始有「可解释、有反馈」的感知基础 |
| P1 | 面试评估闭环与通知/评价在界面上的可见（已完成 → 已通知 → 已出评价） | 闭环感与完成感 |
| P2 | 耗时对比或「约节省 X 小时」的估算展示 | 效率价值可量化 |
| P2 | 候选人端反馈页面或邮件模板 | 公平感与品牌价值 |

## 五、小结

- **业务目标**：初筛效率、决策质量、可配置可审计、面试闭环、候选人体验——功能上多数已覆盖或部分覆盖，但**候选人触达与面试完整闭环仍缺**。
- **痛点解决度**：对 HR（效率）、招聘经理（可解释决策）、面试官（画像与证据）、HR 运营（配置与审计）**在「能用上」的前提下已解决或大部分解决**；对候选人**尚未真正解决**（无反馈、无触达）。
- **价值感不足的主因**：入口错位（HR 用不上）、价值未量化（无节省时间/可解释率等）、候选人无触达、闭环未在体验上显性化。
- **价值感设计**：从「结果摘要一句话」「单一可用入口」「未通过原因摘要与反馈」「闭环与通知可见」入手，再辅以可量化的效率表述和候选人端触点，能让用户明显感受到「痛点被解决」和「产品有真实价值」。
