# 业务回路与全流程观测

## 1. 定位

按 **UAS 进化架构** 构建业务回路：**自主调用蜂群**，对回路的**所有执行过程进行观测**，并基于评估与漂移规则**解决全方位问题**（目的漂移、治理缺口、进化指标、业务运行时一致性）。

## 2. 回路定义

- **配置**：`configs/business_loop_config.json`
- **步骤来源**：`evolution_policy.iteration.default_loop`
  - `intent_activation` → 意图守恒智能体（intent_guard）
  - `governance_check` → 治理审计智能体（governance_guard）
  - `evolution_plan` → 进化规划智能体（evolution_planner）
- **蜂群**：`workflow_config.swarm.agents`，每步映射到对应 agent，执行方式：
  - 若可用 **asui RuntimeManager**：通过 `RuntimeManager.run(topic)` 执行完整 workflow，取该步结果；
  - 否则 **stub_from_plan**：用 `database/plans/ai-full-cycle-recruitment-os.json` 与认知状态生成步骤输出，保证回路可跑通与可观测。

## 3. 观测

- **存储**：`database/observability/loop_runs/{run_id}/`
- **每次运行生成**：
  - `run_summary.json`：run_id、步骤列表、评估状态、漂移动作、治理校验结果
  - `evaluation.json`：进化评估结果（status、risks、suggestions、missing_purpose_keywords）
  - `steps/{step_id}_start.json`：步骤开始时间、agent_id、input_summary
  - `steps/{step_id}_end.json`：步骤结束时间、output_summary、status、error（若有）
  - `drift_events.json`（若发生漂移）：触发的规则与待执行动作

观测覆盖**全流程**：从意图激活到治理校验、进化规划，再到评估与漂移处理，所有节点均有记录，便于排查与复盘。

## 4. 全方位问题与处理

| 检查项 | 说明 | 处理 |
|--------|------|------|
| **purpose_keywords** | 目的守恒：required_purpose_keywords 命中数 ≥ minimum_keyword_hits | 未达则 risks + return_to_purpose_activation |
| **governance_controls** | 治理：平台底座、runtime、治理控制、演化回路 | 调用 evaluate_evolution.py，结果写入 run_summary.governance_check |
| **evolution_metrics** | 进化指标：evaluation_thresholds 与 success_metrics 对照 | 低于阈值则 risks + 对应 rebuild/repair 建议 |
| **drift_handling** | 漂移规则：needs_evolution 时匹配 drift_rules | 记录 drift_events.json，应用 return_to_purpose_activation / rebuild_micro_experience / rebuild_evaluation_rubric / repair_meso_workflow |
| **business_runtime_sync** | 与 examples/ai-recruitment 业务运行时一致（可选） | 可扩展：读取 events.json / candidates 状态做一致性检查 |

## 5. 运行方式

```bash
cd projects/ai-recruitment-os
python scripts/business_loop_runner.py
```

- 退出码 0：所有步骤成功且评估 pass；否则为 1。
- 观测数据始终写入 `database/observability/loop_runs/{run_id}/`。

## 6. 配置项摘要

- `configs/business_loop_config.json`
  - `loop.steps`：本次执行的步骤列表
  - `observation.enabled`：是否写入步骤起止与评估
  - `swarm_invocation.mode`：autonomous
  - `swarm_invocation.fallback_when_no_llm`：stub_from_plan（或 stub_only 强制仅用 stub）
  - `drift_handling.on_needs_evolution`：record_and_apply_rule
  - `full_scope_checks`：purpose_keywords、governance_controls、evolution_metrics、business_runtime_sync

通过调整上述配置，可在不改代码的前提下扩展回路步骤、观测粒度与全方位检查范围。
