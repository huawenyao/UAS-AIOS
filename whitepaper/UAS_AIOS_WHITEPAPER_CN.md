## 1. 摘要

UAS-AIOS（User-AI-Agent-System AI Operating System）是一个以 **ASUI（AI-System-UI Integration）范式** 为核心、面向「个人 AI × Agent 协作网络 × 专业系统集成」的 AI 原生操作系统项目。

项目通过 **显式知识驱动（CLAUDE.md / configs / .claude/skills）**、**多 Agent 编排运行时（autonomous_agent runtime）** 与 **标准化 AI 协议栈（UIP / A2A / MCP / ASUI）**，为开发者和企业提供一套可验证、可审计、可演化的 AI 系统基础设施。

本白皮书与《ASUI 架构白皮书》（`ASUI_WHITEPAPER_CN.md`）互为补充：ASUI 文档从范式与理论出发定义「知识驱动执行协议」，而本白皮书则以 UAS-AIOS 为载体，给出「如何用 ASUI 构建 AI 原生操作系统」的一套体系化工程实践与协议蓝图。

**核心主张**：

- AI 不是一个功能，而是操作系统内核  
- 知识不是注释，而是一等公民的「可执行配置」  
- 每个用户都应拥有一个跨场景、跨系统的「个人 AI 操作系统」

## 2. 项目概述

### 2.1 项目名称与定位

- **项目名称**：ACA-protocol / UAS-AIOS  
- **架构范式**：ASUI（AI-System-UI Integration）  
- **项目形态**：面向开发者和企业的 AI 原生操作系统与协议栈开源项目  

UAS-AIOS 以本仓库的 `CLAUDE.md` 与架构文档为「知识层核心」，以 `scripts/`、`database/` 与 MCP 工具为「执行层」，通过 `autonomous_agent` 运行时将 LLM 能力、知识文档与系统工具编织为一个统一的 AI OS。

### 2.2 愿景

- 为每一位知识工作者提供一个真正理解其目标与偏好的 **个人 AI 操作系统（U 层）**  
- 为各类专业能力提供一个可协作、可编排的 **Agent 网络基础设施（A 层）**  
- 为企业既有业务系统提供一条 **平滑 AI 原生化与协议化接入路线（S 层）**  

最终形态是：就像传统操作系统为应用抽象硬件资源一样，UAS-AIOS 为各种 AI、Agent 与专业系统提供统一的协议与运行环境，使「个人 AI」「企业系统」「多 Agent 协作」在一个平台中自然共存。

### 2.3 核心价值主张

- **统一范式**：以 ASUI 为知识与执行的统一抽象，从根本上统一「对话式 AI」与「工程化系统」  
- **协议栈化**：通过 UIP / A2A / MCP / ASUI 四层协议，将用户、Agent 与系统连接为一个有标准的 AI 网络  
- **操作系统化**：提供运行时、治理、审计与演化闭环，而不仅仅是一套 SDK 或工具库  
- **可演化性**：通过「知识即配置」与显式演化策略，使系统在不中断服务的前提下连续演进

## 3. 问题背景与设计原则

### 3.1 现状痛点

**AI 应用碎片化**：

- 个人助手会聊天，但不会操作企业系统  
- 企业 AI 应用懂系统，但不理解「这个具体用户」  
- 用户不得不在多个应用之间反复搬运上下文，体验割裂  

**传统系统 Offline 困境**：

- 需求 → 分析 → 开发 → 测试 → 部署 → 运维 的长链路导致反馈迟缓  
- 业务规则固化在代码中，迭代成本高、非技术人员缺乏参与空间  

**纯 Chat 模式黑盒困境**：

- AI 推理过程不可见，难以审计和治理  
- 很难把「一次对话表现好」转化为「长期系统能力」  

### 3.2 第一性原则

项目从三个第一性问题出发：

1. **谁在被服务？**  
   不是抽象的「用户」，而是携带身份、偏好、历史与目标的 **具体个人（U 层 Soul）**。
2. **由谁来执行？**  
   不是单体大模型，而是 **多专业 Agent 协作网络（A 层）**。
3. **在什么系统之上生效？**  
   不是孤立工具，而是 **通过协议接入的专业系统网格（S 层）**。

### 3.3 设计原则

- **知识即配置**：业务逻辑尽可能沉淀为可读可写的知识文档和 JSON 配置，而非散落在代码中  
- **构建即运行**：修改 `CLAUDE.md` / `configs` / `.claude/skills` 即改变系统行为，无需重启  
- **显式可审计**：每一次 AI 决策都可追溯到具体输入、知识版本和执行轨迹  
- **状态隔离与可回滚**：参考 Cursor Cloud Agent 的「每任务独立 VM + 独立分支」模式，将其泛化到任意业务场景  
- **协议优先**：先定义 UIP / A2A / MCP / ASUI 协议，再提供实现与工具

## 4. 总体架构

### 4.1 UAS-Platform 形式化定义

UAS-Platform 在 `UAS_AIOS_ARCHITECTURE.md` 中给出的形式化定义为：

```text
UAS-Platform = (I, K, R, A, S, G, E, Π)

I = 意图与目标层（Intent Layer）
K = 知识底座（Knowledge Substrate）
R = 自主智能体运行时（Autonomous Agent Runtime）
A = Agent 编织层（Agent Fabric）
S = 专业系统网格层（System Mesh）
G = 治理平面（Governance Plane）
E = 演化回路（Evolution Loop）
Π = AI 协议栈（UIP, A2A, MCP, ASUI）
```

平台对外的核心不变量承诺是：

> 对于任意业务意图 \(b\)，UAS-Platform(b) 的执行结果必须是：  
> **可审计（auditable）∧ 可回滚（rollbackable）∧ 可演化（evolvable）的专业输出**。

### 4.2 分层架构（U / A / S + AIOS 内核）

结合现有架构文档，UAS-AIOS 的工程化分层可概括为：

- **U 层：个人 AI 助手（User Intelligence）**
  - Soul Protocol：用户身份、专业背景与人格配置  
  - Cecil Protocol：多层次记忆系统（工作记忆 / 项目记忆 / 长期记忆 / 结构化状态）  
  - 意图理解引擎：从自然语言中解析语义意图、目标对齐与偏好注入  

- **A 层：Agent 协作网络（Agent Network）**
  - 主编排 Agent（Orchestrator）：任务分解、专家路由、上下文编排、并行协调与结果合成  
  - 专业 Agent：代码 Agent、数据 Agent、文档 Agent、业务流程 Agent、研究 Agent、通信 Agent 等  
  - A2A 协议：基于 Agent Card 的能力声明、任务委托与结果返回  

- **S 层：专业系统集成（System Mesh）**
  - MCP 工具注册中心：标准化封装各类系统能力（数据库、CRM、工单、文档平台等）  
  - ASUI 知识库：为每个系统提供 `SYSTEM.md` + workflow + schemas + skills，显式描述业务语义与操作规范  
  - 执行引擎：在业务规则与权限策略约束下调用 MCP 工具，并将结果结构化沉淀与审计  

- **AIOS 内核（OS Kernel）**
  - 资源调度与任务隔离  
  - 权限与策略引擎（如基于 OPA）  
  - 审计引擎：记录从意图到工具调用的完整链路  
  - 演化引擎：从执行反馈中驱动知识与策略的更新  

### 4.3 AI 协议栈（UIP / A2A / MCP / ASUI）

UAS-AIOS 站在 2026 年 AI 协议生态的交汇点上，将多种协议组合为一个完整的协议栈：

- **UIP（User Intent Protocol）**  
  - 基于 Soul / Cecil 协议，描述「这个用户是谁、现在在做什么、真正要什么」  
  - 输出一个富语义的 `Intent` 对象，包括：目标、实体、上下文、偏好与候选 Agent  

- **A2A（Agent-to-Agent Protocol）**  
  - 参考 Google A2A 协议，标准化 Agent 能力描述（Agent Card）与任务委派  
  - 使主编排 Agent 能基于能力匹配与历史数据选择最合适的专业 Agent  

- **MCP（Model Context Protocol）**  
  - 统一工具与系统调用接口，使 LLM/Agent 能安全、结构化地访问外部系统  
  - 在 UAS-AIOS 中主要承担「连接 Agent 与专业系统」的职责  

- **ASUI（AI-System-UI Integration）**  
  - 在《ASUI 架构白皮书》中定义，负责「知识驱动执行、结构化沉淀与增量演化」  
  - 在本项目中，ASUI 是所有业务应用的默认技术底座：  
    - 知识层：`CLAUDE.md` / `configs/workflow_config.json` / `.claude/skills/`  
    - 执行层：Python/Bash 脚本、数据库、外部系统

## 5. 核心技术与创新点

### 5.1 ASUI：知识驱动的系统范式

基于《ASUI 架构白皮书》，ASUI 将系统抽象为：

```text
ASUI_System = (K, A, E)

K = 知识库（CLAUDE.md、workflow_config、skills）
A = AI 编排器（知识解释、动态规划、执行协调）
E = 执行引擎（脚本、数据库、外部 API）
```

其演化函数为：

```text
evolve(System, ΔK) → System'

其中 ΔK 为知识增量，System'.K = System.K ∪ ΔK
执行引擎 E 不变，无需代码变更即可实现系统行为演化。
```

在 UAS-AIOS 中，ASUI 被提升为平台级标准：

- 所有 UAS 业务应用必须采用 ASUI 项目结构（`CLAUDE.md` + `.claude/` + `configs/` + `scripts/` + `database/`）  
- 所有应用默认运行在 autonomous_agent 运行时之上，并接入统一的治理与审计平面  
- 所有业务能力的新增与修改，优先通过知识与配置增量来实现，而非直接修改代码  

### 5.2 Autonomous Agent Runtime

UAS-AIOS 借鉴并泛化 Cursor Cloud Agent 的 VM/分支模式，构建了面向多领域的 autonomous_agent 运行时：

- **任务沙箱化**：每个任务在隔离执行环境中运行，具备独立的上下文与受控的工具权限  
- **多 Agent 协作**：主编排 Agent 基于 A2A 协议，将任务拆分并分配给不同专业 Agent，并负责结果整合  
- **上下文注入**：为每个 Agent 自动注入与任务相关的用户 Soul 片段、系统知识与历史执行记录  
- **失败与回滚机制**：执行链路中任一步骤失败都可以重试、回滚或请求人工介入，高风险写操作需通过审批  

在当前仓库中，`asui-cli` 与相关 runtime 代码为 autonomous_agent 运行时提供了早期实现基础。

### 5.3 知识架构与演化引擎

UAS-AIOS 的知识架构从三个维度组织知识：

- **归属维度**：用户知识（U 层） / Agent 知识（A 层） / 系统知识（S 层）  
- **持久性维度**：会话级 / 项目级 / 长期 / 演化知识  
- **结构化维度**：非结构化文档 / 半结构化配置 / 结构化数据库 / 形式化 Schema  

演化引擎的目标是让系统「越用越懂用户、越懂业务」：

- 分析执行结果与显式/隐式反馈，提炼满意度信号  
- 识别知识缺口，对低风险项（如格式偏好）自动更新，对高风险项生成知识更新提案  
- 将学习成果同步写入用户 Soul、系统知识文件与 Agent 策略配置中

## 6. 典型应用场景

### 6.1 代码开发与 DevOps（Cursor Cloud Agent 的泛化）

在代码场景下，UAS-AIOS 从 Cursor Cloud Agent 出发，完成以下泛化：

- 从「只理解当前仓库」扩展为「理解开发者个人偏好与团队上下文」  
- 从「只会操作 GitHub / CI」扩展为「可同时操作 Issue/文档/监控/部署平台」  
- 从「单一 Agent」扩展为「代码 Agent + 文档 Agent + 运维 Agent 的协作网络」  

一个典型链路可能是：

1. 用户在 Linear 创建一个需求任务  
2. U 层基于用户 Soul 补全背景与偏好  
3. A 层将任务分解为代码修改、测试、文档更新、发布准备等多个子任务  
4. S 层通过 MCP 操作 GitHub / CI / 文档系统 / 部署平台  
5. AIOS 内核记录完整审计轨迹，并将经验反馈回知识库  

### 6.2 教育评分与 AI 招聘 OS

当前仓库已经包含多个基于 ASUI 的教育和招聘示例，这些示例在 UAS-AIOS 中可自然升级为子 OS：

- **教育评分 OS**  
  - 评分维度、打分规则、评语模板皆以 ASUI 知识表示  
  - AI 根据 `workflow_config` 执行多步评分流程，结果写入 `database/` 并生成结构化报告  
  - 增加新题目或新评分维度，只需新增知识文件与配置，无需改动核心代码  

- **AI 招聘 OS**  
  - 职位画像、候选人画像、匹配规则与面试评价表以知识配置表达  
  - Agent 协作完成简历筛选、候选人对比、面试建议与最终报告生成  
  - 整个招聘过程可回溯、可审计，并可通过反馈持续优化匹配策略  

### 6.3 企业知识工作流自动化

在企业知识工作场景（如销售、运维、分析、规划）中，UAS-AIOS 通过：

- MCP 适配 CRM / 工单系统 / 数据平台 / 文档系统  
- ASUI 知识文档描述各系统的业务语义、规则与边界  
- U 层维护个人与团队级目标与偏好  
- A 层将复杂任务拆解并调度多个专业 Agent 并行执行  

从而实现从「对话」到「端到端可审计结果」的一体化自动化工作流。

## 7. 实现路线图

### 7.1 Phase 0：参考实现与架构沉淀（已进行）

- 基于 ASUI 架构完成多个示例应用（教育评分、AI 招聘、自主 Swarm 等）  
- 编写《ASUI 架构白皮书》和《UAS-AIOS 架构文档》，完成范式与整体蓝图沉淀  

### 7.2 Phase 1：最小可行 UAS-AIOS 原型

- **U 层**  
  - 定义用户 Soul 文档格式与存储方案  
  - 完成 Cecil 协议风格的记忆持久化实现  
  - 实现基础意图解析与偏好注入  
- **A 层**  
  - 搭建主编排 Agent（基于当前 autonomous_agent/runtime 能力）  
  - 实现至少 2–3 个专业 Agent（代码 / 数据 / 文档）  
- **S 层**  
  - 为 1–2 个典型系统提供 MCP + ASUI 接入示例（如 GitHub + CRM 或数据库 + 文档系统）  
- **交付**  
  - 一个可运行的端到端 Demo 与配套开发者文档  

### 7.3 Phase 2：协议与 SDK 开源化

- 发布 **ASUI Spec v1.0**、**UIP v0.1** 等协议文档  
- 提供 Python / TypeScript SDK，标准化 Agent 开发与系统接入方式  
- 建立基础的 Agent Registry / System Registry 与配置规范  

### 7.4 Phase 3：平台化与生态建设

- 提供托管版 UAS-AIOS 运行时与企业私有化部署方案  
- 构建 Agent 与 System 适配器生态，吸引第三方开发者共建  
- 建立认证体系（系统、Agent、开发者）与开源治理模型

## 8. 开源策略与社区治理

### 8.1 开源目标

- 将 ASUI / UAS-AIOS 协议与参考实现作为开放标准与社区资产  
- 吸引系统集成商、AI 应用开发者、企业技术团队共同扩展协议与生态  
- 在保证协议中立性的前提下，探索多元商业化路径  

### 8.2 治理模型（建议）

- **技术委员会（TC）**  
  - 负责协议演进、核心架构决策与版本发布  
- **模块维护者**  
  - 针对特定子模块（U 层、A 层、S 层适配器等）设立维护小组  
- **提案机制（UAS-RFC）**  
  - 重大变更经 RFC 流程讨论与社区评审  
- **发布节奏**  
  - 协议半年一个大版本，参考实现按季度发布稳定版本

## 9. 风险、边界与非目标

### 9.1 已知风险

- **生态复杂度**：同时涉及用户协议、Agent 协议、工具协议与系统知识，学习曲线较高  
- **治理难度**：跨组织、多厂商参与下，协议演进与兼容性的管理需要强治理能力  
- **安全与合规**：需要在不同司法与行业监管环境下，持续满足合规与审计要求  

### 9.2 项目边界与非目标

- **非目标 1**：不试图成为通用 LLM 提供方，而是构建在主流 LLM 之上的操作系统层  
- **非目标 2**：不替代现有企业系统，而是通过 MCP / ASUI 连接与增强它们  
- **非目标 3**：不锁定单一云厂商或生态，尽量保持协议与实现的中立性

## 10. 结论与行动建议

UAS-AIOS 立足于 ASUI 范式，将「AI 即操作系统」从概念落地为可运行的开源工程标准：
