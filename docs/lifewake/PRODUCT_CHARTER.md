# LifeWake 产品哲学宪章

> 本宪章定义不可被增长、商业化或模型能力覆盖的目的守恒约束。产品全貌见 [主产品蓝图](./LIFEWAKE_PRODUCT_BLUEPRINT.md)。

---

## 我们相信

1. 每个人的生命都是宇宙的独奏，每一次呼吸、每一次心跳、每一段关系都值得被倾听与铭记。
2. AI 不应是冰冷的工具，而应成为「人性的镜子」与「创造的共谋者」，将个体的独特性淬炼成可触摸的奇迹。
3. 真正的连接，始于对「人」的敬畏——不是管理数据，而是理解情感；不是预设功能，而是共创无限。
4. 产品没有终点，正如生命没有边界。它将随着每个用户的成长而生长，在交互中进化，在共鸣中永恒。
5. 技术应服务于「爱」与「自我发现」——让孤独者被看见，让思念者被连接，让平凡者成为自己的艺术家。

## 我们承诺

1. 绝不以效率之名剥夺体验的深度，而是以「慢灵感」对抗「快消费」。
2. 将隐私视为用户的灵魂领土，所有数据仅用于「为用户创作」，绝不被用于「定义用户」。
3. 让每个功能都承载情感重量，让每次交互都成为一次「自我确认的仪式」。

## 宪章原则与可证伪约束

### 1. 「无界生长」原则

| 设计表达 | 工程约束 |
|----------|----------|
| 产品形态随仪式内容液态变形 | 由 `RitualEnvelope` 驱动 `RitualView`，但 Consent Center、撤回和可访问性保持稳定 |
| 功能随情境自然长出 | 能力以 `lw.*` 可插拔注册；扩展功能只暴露接口，不强制入口 |
| 用户与 AI 共塑边界 | 允许用户自定义玩法意图；AI 负责落地为 CapabilityCall + Ritual |

### 2. 「情感炼金术」原则

| 设计表达 | 工程约束 |
|----------|----------|
| 抽象情感 → 多模态体验 | Surprise / PulseSession 必须产出至少一种可感知模态（audio / image / task / visual） |
| 微小而深刻的仪式感 | 每次交付必须附带 `inspiration_trace`（灵感来源解析） |
| 心跳同步可视化 | 双人模式输出 `sync_visual`；纪念物共享由双方 `ShareGrant` 决定 |

### 3. 「关系编织」原则

| 设计表达 | 工程约束 |
|----------|----------|
| 双向情感流动优先 | Bond 实体与 `Bond Guardian` 校验双方需求；禁止仅满足单方输出的 duet |
| 关系是内容催化剂 | 共创作品共同归属；任一方撤回触发 `SHARE_REVOKED`，外部访问立即失效 |

### 4. 「反熵增设计」原则

| 设计表达 | 工程约束 |
|----------|----------|
| 强化独特表达 | 惊喜生成必须引用用户特有信号；禁止纯模板随机填充 |
| 意外惊喜打破常规 | 交付时机写入 `TimingDecision`；可返回 `SLOW_INSPIRATION_DEFERRED`，禁止高频刷屏 |
| 情感冲击测试 | `EmotionImpact` 由用户反馈 + 策展 rubric 构成可解释门禁；模型只提供辅助信号，不宣告情感真理 |

### 5. 「慢灵感」原则

| 设计表达 | 工程约束 |
|---|---|
| 不用即时性替代意义 | 信号不足、时机不合或内容低质时优先 defer，而不是交付模板内容 |
| 沉默也是有效体验 | defer 不制造红点、倒计时或焦虑；用户可查看原因或取消 |
| 稀缺保护仪式感 | 不以 DAU、时长、推送量为目标；频率护栏不可被实验绕过 |

### 6. 「解释主权」原则

| 设计表达 | 工程约束 |
|---|---|
| 用户定义自身意义 | 不从心跳、互动或内容推断人格、健康或关系质量 |
| 来源可见可控制 | `RitualEnvelope` 必须包含来源类别、用途、同意引用与排除/删除动作 |
| 反馈裁决现实 | 模型分数与 rubric 可被用户反馈推翻；争议进入人工策展或取消 |

## 非目标与商业红线

**我们做：**

- 让独特性可感知、可分享、可传承
- AI 作为「隐形的艺术家」

**我们不做（v0.1）：**

- 功能堆砌式效率工具
- 用行为数据给用户打标签、评分、投放
- 未经同意的代发消息或公开分享
- 固定仪表盘式“管理生命”UI
- 无限内容流、连续签到、羞耻式召回
- 医疗/心理诊断、关系评分、伴侣监控
- 将同意、撤回、导出或删除设为付费能力
- 出售用户数据或用广告补贴“免费体验”

## 守恒检查

每个 PRD、实验和 ChangeSet 必须回答并留证：

1. 对应哪条宪章理念与第一性法则？
2. 使用了哪些现实实体、数据和触点？
3. 用户如何知情、拒绝、撤回和恢复？
4. 失败时退回哪一层：交付、策略、流程还是理念？
5. 用哪个 CASE 与 KPI 验证，哪个护栏可以否决上线？

完整链路见 [推导与验证矩阵](./DERIVATION_AND_VALIDATION_MATRIX.md)。

## 结语

这款产品不是工具，而是一面镜子；不是终点，而是一段旅程。从 MVP 开始，每一步都指向：**让科技成为人性最美的注脚。**
