# 企业级数字人生态：SelfPaw × ΠPaw 业务规划与产品能力定义

> **版本**：v1.0 · 2026-05-23  
> **定位**：从「个人私有 / 公网数据」Agent 跃迁到 **企业级数字人生态** 的产品蓝图  
> **架构范式**：类人 AI Agent 为核心 · **平台 + 模型** · 企业业务系统 + 数字化基础能力 · 经营向外  
> **关联文档**：`UAS_AIPOS_SelfPaw_Integrated_Product_Tech_Architecture.md`、`UAS-AIOS_Delivery_Form_And_Product_Design.md`

## 0. Executive Summary

当前主流 AI Agent（OpenClaw、Cursor、ChatGPT 等）的默认场景是：

- **个人私有空间**：本地文件、个人记忆、单操作者信任边界  
- **公网开放数据**：检索、公开 API、无租户隔离的 RAG  

**企业级数字人生态** 需要的是第三条路径：

```
企业主权数据流 + 业务能力服务化 + 类人 Agent 编排
        ↓
  对内：员工数字分身（SelfPaw / User AGI）
  对外：经营数字岗位（ΠPaw / Business AGI）
        ↓
  经营向外：获客 · 履约 · 服务 · 协同 · 决策
```

**产品公式**：

```
企业级 AI 产品 = UAS 平台（Kernel + Studio）
               × 模型层（多 Provider / 领域模型）
               × 数字基础能力（流程 / 表单 / 权限 / 数据）
               × 业务系统连接器（CRM / BPM / 财务 / ERP）
               × 类人 Agent（决策 · 流程 · 调度）
```

## 1. 问题定义：为什么个人 Agent 模式无法直接进企业

| 维度 | 个人/公网 Agent | 企业级数字人要求 | 缺口 |
|------|-----------------|------------------|------|
| **数据** | 用户自有 / 公网爬取 | 租户隔离、分级授权、主数据 | 无企业数据平面 |
| **身份** | 单用户 | 组织·部门·岗位·角色·委托 | 无组织身份模型 |
| **执行** | 工具调用即授权 | L1/L2/L3 分级 + 审批 + 审计 | 无业务门禁 |
| **流程** | 对话式即兴 | BPM 状态机 + SLA + 异常升级 | 无流程编排 |
| **价值** | 个人效率 | 营收、成本、合规、客户体验 | 无经营指标闭环 |
| **对外** | 可选 | **经营向外是核心 KPI** | 无客户/伙伴触达通道 |

**结论**：企业级产品必须先建设 **数据流 + 能力服务化**，再在其上挂载类人 Agent；Agent 不是外挂聊天框，而是 **可执行业务角色的数字人**。

## 2. 第一性原理：企业级数字人生态

### 2.1 数字人三层定义

```
┌─────────────────────────────────────────────────────────────┐
│  L3 经营数字人（对外）                                        │
│  销售顾问 · 客服专员 · 方案顾问 · 渠道伙伴 · 投标专员          │
│  → ΠPaw Business AGI · 经营向外                               │
├─────────────────────────────────────────────────────────────┤
│  L2 职能数字人（对内跨部门）                                   │
│  招聘官 · 财务分析 · 运营策划 · 合规审查 · 项目经理            │
│  → ΠPaw 多 Agent 工作台 + 蜂群/三维推演                       │
├─────────────────────────────────────────────────────────────┤
│  L1 个人数字分身（对内个人）                                   │
│  员工的第二大脑 · 日程/文档/任务代理 · 向上汇总意图与证据       │
│  → SelfPaw User AGI · 组织内嵌实例                            │
└─────────────────────────────────────────────────────────────┘
           ▲                    ▲                    ▲
           └────────────────────┴────────────────────┘
                    共享：UAS Kernel + 企业数据平面
```

- **模型负责理解与生成**；**平台负责执行与保障**（ASUI：Agent 负责规划，确定性引擎负责执行）

- **SelfPaw 与 ΠPaw 共享平台**，差异在 **角色边界、数据 scope、对外通道**

## 3. 双轨产品定位（SelfPaw vs ΠPaw）

### 3.1 定位对照

| | **SelfPaw（User AGI）** | **ΠPaw（Business AGI）** |
|---|-------------------------|---------------------------|
| **本质** | 组织内 **员工数字分身** | 组织 **经营数字岗位体系** |
| **用户** | 每位知识工作者 | 业务负责人、运营、管理者 |
| **数据 scope** | 个人授权 + 岗位授权 | 组织全局（按角色分级） |
| **主要场景** | 对内效率：写、查、办、汇 | 对外经营：获、转、服、续 |
| **Agent 形态** | 单用户 ReAct + 个人蜂群 | 多岗位 Agent + 跨部门编排 |
| **自主级别** | On loop → Above loop（个人任务） | Above loop（流程级自主） |
| **与平台关系** | 组织部署的 **边缘实例** | 组织 **中枢指挥层** |

### 3.2 协同关系（非两套产品）

```
员工 SelfPaw ──意图/证据──▶ ΠPaw 职能 Agent ──能力调用──▶ 业务系统
     ▲                              │
     └──────── 任务回执 / 审批 ◀─────┘

对外客户 ──通道──▶ ΠPaw 经营数字人 ──编排──▶ CRM/BPM/财务
                        ▲
              SelfPaw 员工可介入升级（Human-in-the-loop）
```

**关键设计**：SelfPaw 企业版 **不是** C 端独立 App 的简单 B 端拷贝，而是 **带组织身份的个人 Agent**；ΠPaw 是 **组织级 Agent 编制与调度系统**。

## 4. 企业数字化基础能力（平台必建层）

在挂载类人 Agent 之前，平台需提供 **可产品化封装** 的四类基础能力：

### 4.1 能力矩阵

| 基础能力 | 产品化封装 | Agent 可调用语义 | 对接系统 |
|----------|------------|------------------|----------|
| **流程 Process** | 流程模板 + 状态机 + SLA | `start_process`, `advance`, `escalate` | BPM / 自研 workflow |
| **表单 Form** | 动态表单 + 校验 + 版本 | `render_form`, `submit`, `validate` | 低代码 / NocoBase 类 |
| **权限 Permission** | RBAC + ABAC + 数据 scope | `check_permission`, `delegate` | IAM / 企业目录 |
| **数据 Data** | 主数据 + 实体 CRUD + 事件 | `query_entity`, `emit_event`, `sync_mdm` | CRM / ERP / 数据湖 |

### 4.2 业务能力服务化（Capability Service Catalog）

将 CRM/BPM/财务等 **封装为语义服务**，而非暴露原始 API：

```yaml
# 示例：能力服务契约
service: cs.opportunity
version: v1
operations:
  - name: qualify_lead
    input: { lead_id, criteria_ref }
    output: { qualified, score, evidence[] }
    gates: [G6_compliance, G4_measurable]
    side_effects: [crm.lead.update, event.lead.qualified]
  - name: create_quote
    input: { opportunity_id, product_bundle }
    approval: L2  # 超阈值 L3
```

**目录结构（建议）**：

| 域 | 能力服务 | 来源系统 |
|----|----------|----------|
| 客户 | cs.customer, cs.lead, cs.opportunity | CRM |
| 销售 | cs.quote, cs.contract, cs.order | CRM + 合同 |
| 服务 | cs.ticket, cs.sla, cs.feedback | 客服 / ITSM |
| 流程 | cs.approval, cs.process, cs.task | BPM |
| 财务 | cs.invoice, cs.payment, cs.budget | 财务 / ERP |
| 人力 | cs.recruit, cs.onboard, cs.performance | HRM |
| 协同 | cs.notify, cs.calendar, cs.doc | OA |

Agent **只调用 cs.\***，平台负责 **翻译、权限、审计、重试**。

## 5. SelfPaw（User AGI）产品定义

### 5.1 产品一句话

> **组织内每位员工的类人 AI 分身**：理解个人目标与岗位上下文，在授权边界内完成「理解 → 规划 → 执行 → 汇报」，并将高价值意图升级至 ΠPaw。

### 5.2 企业版 vs 个人版

| 能力 | 个人版（C 端愿景） | **企业版（优先交付）** |
|------|-------------------|------------------------|
| 身份 | 单用户 | 租户 + 部门 + 岗位 + 汇报线 |
| 数据 | 本地 / 个人云 | 企业数据平面 + 个人工作区 |
| 记忆 | 个人 MEMORY | 个人记忆 + **岗位知识 scope** |
| 工具 | 通用 Skills | Skills + **企业 Capability Services** |
| 通道 | 个人 IM | 企业 IM + Console + **Embedded in OA** |
| 治理 | 自行负责 | G1-G7 + L1/L2/L3 + 审计 |

### 5.3 核心能力模块

#### M1 组织身份与上下文绑定

- 登录与企业目录同步（AD / 飞书 / 钉钉）  
- `UserContext` 扩展：`tenant_id`, `dept_id`, `role_ids`, `data_scopes`  
- RuntimeContext 注入 **岗位世界模型片段**（非全量企业本体）

#### M2 个人意图工作台（Intent Hub）

- 自然语言 → `IntentObject`（目标、约束、紧急度、关联 Project）  
- 意图分类：个人效率 / 协作请求 / **需组织 Agent 接管的经营事项**  
- 一键「提交至 ΠPaw」：带 Evidence 的意图单

#### M3 岗位能力包（Domain 企业化）

- 预置岗位 Domain：销售、客服、研发、HR、财务…  
- Domain = 岗位 Ontology + 可用 Capability Services + 话术/合规策略  
- 员工可叠加 **个人 Project**（如「Q2 大客户攻坚」）

#### M4 执行与代理

- 授权内自动执行：查数据、填表单、发起流程、写文档、约会议  
- **数字分身代理**：代拟回复、代填周报、代跟内部流程（需 L2 确认）  
- 跨应用编排经 Capability Service，禁止 Shell 直连生产库

#### M5 认知蜂群（个人决策）

- 五视角 Agent：机会 / 风险 / 执行 / 合规 / 用户价值  
- 用于 **个人重要决策**（转岗、大额采购、方案选型）  
- 输出可审计的 **决策备忘录** → Asset

#### M6  upward 汇总（与 ΠPaw 接口）

- 周期性摘要：本周意图、完成、阻塞、需组织资源  
- 异常升级：SLA 临近、审批卡住、客户投诉 → 自动创建 ΠPaw Working Task

### 5.4 SelfPaw 典型场景

| 场景 | 用户 | Agent 行为 | 调用的能力服务 |
|------|------|------------|----------------|
| 晨间 briefing | 销售 | 汇总昨日线索、今日会议、待办报价 | cs.lead, cs.calendar |
| 客户邮件代拟 | 销售 | 按 CRM 上下文生成回复，L2 发送 | cs.customer, cs.opportunity |
| 报销发起 | 全员 | 识别发票、填表单、提交流程 | cs.expense, cs.approval |
| 研发任务分解 | 工程师 | 关联 Project，创建子 Task，同步 Jira | cs.task, cs.doc |
| 升级客诉 | 客服 | 判断 SLA，升级 ΠPaw 服务 Agent | cs.ticket, cs.escalate |

### 5.5 SelfPaw 分阶段交付

| 阶段 | 目标 | 交付 |
|------|------|------|
| **S1** | 组织身份 + 企业 Console | SSO、租户、岗位 Domain、审计 |
| **S2** | 能力服务接入 | 5 个 cs.\*（OA/CRM 各 2-3） |
| **S3** | 意图升级 ΠPaw | Intent Hub + 跨产品 Working Task |
| **S4** | 分身代理 | L2 邮件/文档/流程代理 |
| **S5** | 个人蜂群决策 | 五视角 + 决策 Asset |

## 6. ΠPaw（Business AGI）产品定义

### 6.1 产品一句话

> **企业经营的类人 AI 岗位体系**：以「平台 + 模型」整合 CRM/BPM/财务与数字基础能力，实现 **经营向外** 的获客、转化、履约、服务与决策闭环。

### 6.2 「经营向外」目标分解

```
经营向外 = 对外触达 × 对内履约 × 实时决策 × 资源调度

  触达：多渠道数字人接待（IM/官网/邮件/伙伴 API）
  履约：订单·合同·交付·开票·回款 流程自主推进
  决策：商机评估·定价·库存·信用·风险 类人判断 + 规则兜底
  调度：人力·库存·产能·预算 跨系统资源编排
```

### 6.3 数字岗位编制（Business Agent Roster）

| 岗位 Agent | 对外/对内 | 核心职责 | 类人能力 |
|------------|-----------|----------|----------|
| **市场洞察 Agent** | 对内 | 竞品、舆情、行业政策 | 趋势判断、机会识别 |
| **线索培育 Agent** | 对外 | 官网/活动线索识别与培育 | 沟通、资格判断 |
| **销售顾问 Agent** | 对外 | 需求诊断、方案推荐、报价协同 | 顾问式销售、异议处理 |
| **投标 Agent** | 对外 | 标书解读、响应生成、合规检查 | 专业写作、风险识别 |
| **客服 Agent** | 对外 | 咨询、工单、SLA | 共情、问题路由 |
| **履约 Agent** | 对内/外 | 订单跟踪、交付协调 | 流程推进、异常升级 |
| **财务 Agent** | 对内 | 对账、开票、信用、预算 | 精确计算、合规 |
| **合规 Agent** | 对内 | 合同/数据/行业规则审查 | 红线识别、拦截 |
| **调度 Agent** | 对内 | 跨部门任务、资源冲突 | 优先级、博弈协调 |
| **经营分析 Agent** | 对内 | KPI、归因、预测 | 数据叙事、建议 |

岗位 Agent **不是** 单一 Chatbot，而是：

```
Domain（岗位认知包）
  + Project（经营目标，如「华东 Q2 营收 1.2 亿」）
  + Working Task（具体实例，如「跟进客户 A 报价」）
  + Capability Services（可执行手）
  + Gate（L3 硬闸）
```

### 6.4 核心能力模块

#### P1 组织世界模型（Enterprise World Model）

- 五维建模：客户（客体）· 组织（主体）· 订单（客体）· 时间（SLA）· 反馈（NPS/回款）  
- 法则库：定价规则、信用政策、审批矩阵、行业合规  
- **镜像 → 透镜 → 熔炉**：反映经营现实 → 看本质矛盾 → 重塑流程

#### P2 多 Agent 协同工作台

- 岗位 Agent 编制、编制上限、成本配额  
- Agent 间 **A2A 协议**：委派、协商、共识（蜂群 / 三维理念现实模板）  
- 人类 **指挥席**：Override、审批、策略调整  
- Task Panel：**任务态**（运行/阻塞/待批/失败/恢复）

#### P3 经营流程引擎（Business Process AGI）

- BPM 语义层：不是画流程图，而是 **Goal → Process Template → Agent 绑定**  
- 动态适配：同一「销售闭环」模板，按行业 Domain 替换步骤与能力  
- 异常升级：Agent 无法决策 → SelfPaw 指定负责人 → 或人工指挥席

#### P4 对外数字人通道（Outward Channels）

- 统一 **Outward Gateway**：官网 widget、企微/钉钉对外、邮件、伙伴 Webhook  
- 每个通道绑定 **岗位 Agent + 租户 + 合规策略**  
- 会话 **全量审计**；PII 脱敏；跨境数据策略

#### P5 业务价值闭环（经营 KPI）

- KPI 绑定 Project：营收、转化率、SLA、回款周期、人效  
- Agent 行动 → **归因到 KPI**（非仅 token 统计）  
- DIKWP 仪表盘：Data → Information → Knowledge → Wisdom → Purpose

#### P6 合规与治理中心

- 租户隔离、数据分级、跨境  
- L1 自动 / L2 确认 / L3 禁止或双人审批  
- ChangeSet 全链路：谁、何时、何 Agent、何能力、何数据、何结果

### 6.5 「经营向外」端到端场景

**场景：B2B 线索 → 签约 → 回款**

```
1. [对外] 线索培育 Agent：官网留资 → qualify_lead → CRM
2. [对外] 销售顾问 Agent：需求诊断 → 方案 → create_quote
3. [对内] 合规 Agent：合同条款 G6 审查
4. [对内] 审批流程：超 50 万 L3 → 人类指挥席
5. [对内/外] 履约 Agent：订单 → 交付 → cs.invoice
6. [对内] 财务 Agent：对账 → 回款提醒
7. [分析] 经营分析 Agent：归因到「华东 Q2」Project KPI
8. [演化] 失败案例 → ChangeSet → Domain 法则更新
```

### 6.6 ΠPaw 分阶段交付

| 阶段 | 目标 | 交付 |
|------|------|------|
| **P1** | 单岗位标杆 | 客服 Agent + cs.ticket + 对外 IM |
| **P2** | 销售闭环 | 线索→报价→合同 + CRM 连接器 |

## 7. 类人 Agent 核心能力设计（跨 SelfPaw / ΠPaw）

### 7.1 类人三能力

| 能力 | 定义 | 工程实现 |
|------|------|----------|
| **业务决策** | 在规则与证据约束下做选择 | 世界模型法则 + 蜂群推演 + Gate |
| **流程处理** | 推进状态机、处理异常、升级 | BPM 语义 + AEE Plan→Act→Verify |
| **资源调度** | 人、物、财、时的最优分配 | 调度 Agent + cs.\* + 约束求解 |

### 7.2 认知循环（统一）

```
Sense（感知经营信号）
  → Bind（绑定组织/岗位上下文）
  → Route（Domain + 岗位 Agent 路由）
  → Plan（ExecutionPlan + 能力链）
  → Act（Capability Service 调用）
  → Verify（Gate G4/G6 + 业务规则）
  → Learn（Feedback → ChangeSet → E 层）
```

### 7.3 灵活动态业务适配（产品化封装）

| 封装单元 | 内容 | 适配方式 |
|----------|------|----------|
| **行业 Domain 包** | Ontology + 话术 + 合规 + Skills | 切换 Domain 即切换行业 |
| **流程模板包** | BPM 定义 + Agent 绑定 + SLA | 参数化步骤与审批矩阵 |
| **能力连接器包** | CRM/BPM/财务 adapter | 配置 endpoint + 字段映射 |
| **法则包** | 定价/信用/审批规则 | 热更新，无需改代码 |
| **岗位 Agent 包** | Prompt + 工具白名单 + KPI | 编制即部署 |

**适配逻辑**：新业务 = 选 Domain 包 + 流程模板 + 连接器 + 法则微调 → UAS Studio 发布 → 运行时热加载（知识即配置）。
